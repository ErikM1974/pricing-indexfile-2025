Vendor rules — the OnSite server is LIVE production

  • Never blanket-query. Always filter (WHERE on a stored, indexed field) + FETCH FIRST n ROWS ONLY. Unbounded scans can fill the server's C: drive and crash OnSite for everyone.
  • Never poll tightly. Automated queries run minutes apart, not seconds. One connection, sequential queries — the xDBC listener serializes work.
  • DSN points at Data_ODBCMapping only. Never other files, never modify it, never create extra DSNs.
  • Resetting the ODBC connector (FileMaker Server Admin Console) disconnects ALL ODBC clients — including the shipping machines. Coordinate before touching it; shipping software must be restarted after.
  • No images/attachments via ODBC (design thumbnails, PDFs). No writes ever — the extro account is read-only; order creation stays on the ManageOrders push API.
  • Need a table/field that isn't mapped? Full-screen OnSite screenshots with fields highlighted → support@shopworx.com. They annotate table/field names and extend the mapping. (Known unmapped: LinesPur — PO line items.)

How we connect bandit bridge · dev

laptop (chat / scripts) → WinRM → BANDIT (192.168.10.219 · drivers + 3 System DSNs: SW32, SWAddress, SWODBC) → ODBC :2399 → OnSite server (192.168.10.6)

The FileMaker driver and the querying app must live on the same Windows machine — bandit fills that role (always-on; already had both drivers from the shipping integration). Remote-query pattern from an authorized laptop:

$cred = Import-Clixml "$env:USERPROFILE\bandit-cred.xml"
Invoke-Command -ComputerName bandit -Credential $cred -ScriptBlock {
  $c = New-Object System.Data.Odbc.OdbcConnection `
    'DRIVER={FileMaker ODBC};Server=192.168.10.6;Database=Data_ODBCMapping;UID=extro;PWD=extro'
  $c.Open()
  # ... bounded SELECT here ...
  $c.Close()
}

Production plan (not yet built): scheduled sync agent on bandit → proxy bulk-upsert route → Caspio tables + freshness watchdog. Heroku can never reach the DSN directly (no Linux driver; LAN-only address). Canonical doc: memory/SHOPWORKS_ODBC_INTEGRATION.md.

FileMaker SQL — rules & traps read before writing queries

  • Row limits: FETCH FIRST 100 ROWS ONLY (no LIMIT/TOP). Paging: OFFSET n ROWS before it — but prefer delta-sync on date_Modification.
  • Dates: {d '2026-07-16'} / {ts '2026-07-16 14:00:00'}. Never BETWEEN on dates — pathologically slow; use >= AND <=.
  • Empty string = NULL. col = '' matches nothing → use IS NULL / IS NOT NULL.
  • Case-sensitive strings. 'a' <> 'A'. LOWER() works but kills index use.
  • Booleans are 1/0 — no TRUE/FALSE keywords.
  • Joins: INNER + LEFT OUTER only (no RIGHT/FULL — rewrite as LEFT).
  • ⚠ THE BIG ONE — calc fields: prefixes cn_ ct_ cd_ cm_ co_ cnCur_ cnPer_ sum_ are unstored calculations/summaries — evaluated per-row, unindexable. Never in WHERE; avoid in SELECT. One calc field can turn a sub-second query into minutes of server CPU. gn_ gt_ gd_ gm_ go_ = globals (session values — useless to us). Plain date_ cur_ sts_ ID_ id_ fields are stored → fast.
  • Never SELECT * — explicit column lists only (wide tables evaluate every calc per row).
  • Schema discovery: SELECT TableName, FieldName, FieldType FROM FileMaker_Fields (.NET GetSchemaTable() overflows on wide tables — FM15 driver bug).
  • Repeating fields (decimal[14], varchar[10]) need special handling — plain SELECT returns repetition 1.
  • Expect dropped connections. The fmxdbc_listener process can wedge; set timeouts, retry once, then alert.
  • Multi-state status codes: sts_* = 0 No · 1 Yes · .5 Partial · 8/222 N/A (same as ManageOrders).
