The headline: v4 is built for SQL + AI

v4 adds bulk operations with safety rails, full T-SQL in query parameters, single-call schema discovery, first-class file/webhook/app endpoints, and a machine-readable "AI context" so agents can drive the API. Our proxy still runs v3 — adopt v4 per-endpoint only where a new power pays off.

What's new in Caspio 73.0 released 2026-07-02

The v4 launch shipped as part of Caspio 73.0. Several 73.0 changes are platform-level and matter beyond the endpoint list — some affect our existing v2/v3 integrations. Full notes: howto.caspio.com/release-notes/caspio-73-0.

Granular permissions now enforced — impacts v2 AND v3, not just v4

REST endpoints now return 403 Forbidden when the calling API profile lacks object-level permission (Data Import/Export Tasks, Bridge Apps); Tables password-field & attachment reads now return 404 for missing objects. Action: confirm our proxy's API profile has "Enable access to all objects" ON (Integrations → API Profiles) so the ~397 proxy calls don't silently start 403-ing, and treat a 403 as an auth-scope error (surface it — never swallow).

HMAC-SHA256 signature on every outgoing webhook

Each outgoing webhook now carries an HMAC-SHA256 signature (secret = the webhook's autogenerated secret / regenerateSecret) so a receiver can prove it came from Caspio, unmodified. Compatibility mode for receivers that can't compute an HMAC: include the secret in the payload (defaultPayloadIncludesSecret / [@out-hook:secret]). Capture the exact signature header from the authenticated v4 Swagger when we build the receiver.

New egress IPs for webhooks + import/export tasks

If any receiver or firewall allowlists Caspio's egress IPs, add the new ones (all /32 IPv4, US set): 3.94.26.149, 3.221.12.171, 3.223.237.156, 3.232.188.127, 18.235.29.217, 34.200.156.163, 35.173.3.24, 44.198.154.72, 44.219.25.193, 44.221.229.56, 54.88.136.246, 54.224.65.145, 100.58.20.55. The list may change without notice. (NWCA's 24 webhooks target Zapier today, so no NWCA-owned allowlist is affected yet.)

Also in 73.0

  • Redesigned API Profile Management (Integrations → API Profiles) — granular default permissions by object type + per-object overrides, generate a bearer token from the UI and test in the built-in Swagger UI, and IPv6 + IP-range access control. This is the path to split our single-credential blast radius into scoped least-privilege profiles.
  • Flex — Theme Editor (per-Segment themes, live preview, reusable) + Date/Time minute-level min/max offsets and fixed daily time windows.
  • Vault — create app packages directly from a Flex application's menu.

03 Authentication OAuth 2.0 client credentials

All data endpoints require Authorization: Bearer {access_token}. Auth is identical to v3 — same token endpoint and lifecycle.

POST {tokenEndpoint}          # account subdomain, same host family as the API base
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id={clientId}&client_secret={clientSecret}

→ { "access_token":"…", "token_type":"bearer", "expires_in":86399, "refresh_token":"…" }
  • Cache & reuse access_token; renew after expires_in (~24h). Refresh without re-sending creds: grant_type=refresh_token&refresh_token={…}.
  • Each API profile has an isolated permission boundary scoped to specific tables/views — an application-level allowlist enforced before any SQL runs. Out-of-grant references are rejected as IncorrectQueryParameter.
  • Our implementation: caspio-pricing-proxy/src/utils/caspio.js → getCaspioAccessToken() (module-cached, 60s buffer). 400 "IncorrectQueryParameter" = stale token, auto-retried once per page.

04 Querying records the T-SQL power feature

Query parameters are plain camelCase (no q. prefix):

ParamMeaning
selectComma field names, *, or any valid T-SQL SELECT expr — aggregates (COUNT, SUM…), scalar subqueries, CASE, arithmetic.
whereT-SQL WHERE (no keyword). EXISTS, IN + subqueries, compound conditions across any permitted table.
orderBye.g. LastName ASC, FirstName DESC.
groupByFields/exprs; supports HAVING, e.g. Status HAVING COUNT(*) > 5.
limit1–1000, default 100. Legacy; ignored when pageNumber/pageSize set.
pageNumber / pageSizeExplicit paging. pageSize 5–1000 (default 25 when paging).

Rules that bite

  • Always reference a field by its Name, never its Label.
  • Strings need the Unicode prefix in where: N'value'. Escape single quotes by doubling: N'O''Brien'.
  • YES/NO fields: use 1/0 in where, but send JSON true/false in write payloads.
  • Cross-table subqueries are permitted & encouraged; the permission boundary still applies. System catalogs are blocked and DDL (CREATE/ALTER/DROP) is never allowed.

Why it matters for us

Rich SQL-driven analytics straight through REST — no stored procedures, middleware, or pre-built Caspio views. Reporting aggregation (e.g. daily-sales-by-rep) could move server-side.

05 Write operations & response envelope

List endpoints return an envelope — read records from data[], use pagination.totalCount to compute pages:

{ "data": [ … ], "pagination": { "totalCount": N, "pageNumber": N, "pageSize": N } }
VerbBehavior
POST …/recordsInsert one. Omit auto-generated (Editable-false) fields. Returns 201 + new PK_ID (or full record with echo=true).
PATCH …/records/{pkId}Partial update — only supplied fields change; pass null to clear a field.
PUT …/records/{pkId}Update by PK_ID (symmetric with PATCH by design).
DELETE …/records/{pkId}Delete by PK_ID.

Every write endpoint accepts an optional echo=true query parameter to return the affected record(s).

06 Bulk operations & safety guardrails the big new capability

Bulk insert takes a JSON array (max 1000) and returns 201 when all succeed or 207 Multi-Status on partial failure — each item carries its own status:

POST /v4/tables/{tableId}/records/bulk
[ { "FieldA": 1 }, { "FieldA": 2 } ]

→ { "PK_ID": ["101","102"],
    "data": [ { "status":201, "PK_ID":"101" }, { "status":422, "error":"…" } ] }

Conditional bulk update & delete require a where:

  • BulkUpdateRecordsRequest — where * (required) + recordValues * (required)
  • BulkDeleteRecordsRequest — where * (required)

Tautology guard

An always-true filter (1=1, PK_ID>0) would hit every row — irreversible for DELETE. When the server's tautology guard is active, such unscoped filters are rejected. Always target with something like PK_ID IN (1,2,3) or Status=N'Inactive', and run a SELECT with the same where first to confirm the row count.

07 Error responses actionable diagnostics

{ "Code":"ERROR_CODE", "Message":"human description", "Resource":"/path",
  "RequestId":"…", "DocumentationUrl":"…",
  "Hint": { "Note":"…", "Remediation":"…" } }

Every error links back to /v4/errors/{code} for a machine-readable hint + remediation. Structural deletes (deleting a table, folder, or Flex app) return 405 by design as schema protection — not a missing feature.

08 Data model — types, fields, schema, relationships

Data types (exact token strings)

Fields flagged Editable=false are auto-generated and must never appear in POST/PUT/PATCH bodies.

No (auto)PK_ID AUTONUMBER SEQUENCE RANDOM_ID GUID TIMESTAMP FORMULA
LegacyFILE — read-only path field in v4; may appear on older tables

Field model & schema discovery

  • POST/PATCH …/fields use FieldModelV4 (name*, dataType*, label, displayOrder, unique, isFormula, format, prefix, length, listField[], stampOnInsert/Update+stampTimeZone, attachment controls). Reads return richer FieldItemV4 (+editable, formula, relationship).
  • One call, everything: GET /v4/schemas/tables (also /views, /directories, /outgoingWebhooks) returns every accessible object with full field definitions, relationships, and triggered actions. Ideal when object names are unknown — kills field-list hard-coding.
  • Relationships are explicit (TableRelationshipItem: relationshipType, joinType, parent/child table+field, referentialIntegrity, cascadeUpdate/Delete, displayValue) — see joins without reverse-engineering views.

09 Files & attachments two distinct surfaces

File Library — account-level storage under /v4/fileAssets

  • List / search files & folders, upload (POST new, PUT overwrite), download or delete by fileId (GUID) or by path.
  • Metadata: fileId, name, size, contentType, fullFilePath. New-file upload returns 409 Conflict if the name already exists.
  • Typical FILE-field flow: stored path → GET /v4/fileAssets/files/path/fileInfo (resolve the fileId) → GET …/files/{fileId} (download).

Attachment fields — per-record binary on tables / views / directory users

  • Download / upload-replace (PUT) / delete a single attachment by record + field name.
  • Bulk attachment ops by condition: upload to many records, get/rename file metadata, delete matching files.

10 Webhooks & automation

Full CRUD for outgoing webhooks and their events, plus PATCH …/{webhookId}/regenerateSecret. Events use WebhookEventV4PostRequest:

WebhookEventV4PostRequest {
  type *          string     // required
  objectId *      string     // required — the table/directory
  objectFields    string[]
  eventSources    string[]
  defaultPayloadIncludesSecret  boolean
  defaultPayloadFields          string[]
  customPayloadTemplate         string
  enabled         boolean
}

Triggered-action rules surface (read-only) through schema discovery as TriggeredActionItem (name, eventType, isEnabled, referencedObjects[] + audit stamps).

NWCA note

Caspio Outgoing Webhooks FIRE on REST writes (≠ Triggered Actions, which do NOT fire on REST). v4 gives full webhook CRUD — reinforces the roadmap to convert polling crons → authenticated push. Guard against feedback loops (scope eventSources/objectFields, idempotent dedupe).

11 Apps, DataPages & tasks new in v4

  • Flex applications — list & read (AppItemV4: appId, name, dataPageCount, icon, audit stamps).
  • Bridge applications — list/read apps, list their DataPages, read a DataPage, get its deployment code, and deploy/undeploy one or (in bulk) all DataPages.
  • Data Import/Export tasks — list, read, and POST …/{taskId}/run. ScheduledTaskInfoV4 exposes status, recurrence, last-run time/result, and time zone.

12 Proxy adoption — the v3 → v4 migration path

Current state: caspio-pricing-proxy calls Caspio v3 exclusively. The single migration chokepoint is caspio-pricing-proxy/src/utils/caspio.js (token cache, fetchAllCaspioPages(), base URL https://c3eku948.caspio.com/integrations/rest/v3).

High-value v4 wins for us

  1. Bulk conditional writes instead of per-record loops (respect the tautology guard; SELECT-first).
  2. Single-call schema discovery to stop hard-coding field lists.
  3. T-SQL where/select for analytics without pre-built Caspio views.
  4. Outgoing-webhook CRUD to replace polling crons (webhook-fires-on-REST).
  5. /v4/aiManifest + lenses for AI-driven tooling.

Migration checklist (per endpoint — do NOT bump the whole repo)

Drop the q. prefix → read from data[] not Result[] → use pageNumber+pageSize → N'…' string literals + 1/0 in where (but true/false in bodies) → confirm the token's profile grants the referenced objects. v3 works today; migrate an endpoint only when a specific v4 power justifies it.
