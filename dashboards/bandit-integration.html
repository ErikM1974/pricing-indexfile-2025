Bandit Integration how ShopWorks data & thumbnails flow to Caspio & Box

Admin Dashboard

Bandit is the always-on Windows box on the shop LAN that carries ShopWorks data into Caspio. It is not a file-copy or a Box client — it runs small PowerShell agents on a schedule that read ShopWorks and POST to our proxy, which does every Caspio write and every Box upload. Living document — kept current whenever the bandit tasks or sync scripts change. Verified live 2026-07-18.

The one rule that bit us

Never run two write-heavy Caspio syncs at full speed at the same time. On 2026-07-17 the thumbnail-metadata sync and image sync ran concurrently and tripped Caspio's per-second rate limit (api-calls-rate) — even though the monthly Integrations meter was only 496K/500K. The monthly cap is soft (billed overage), not a hard block; the per-second burst limit is the real ceiling. It looked like an outage but recovered the instant the concurrency stopped.

Steady-state, not backfill — keep them from bursting together

These are gentle now (deltas of a few rows/images per run). The one guard: don't let the metadata + image syncs fire a large batch at the exact same second (that's the per-second-limit trap). Steady-state overlap is fine; a future bulk re-load should be staggered or done via CSV import.

Common jobs

  • Is a sync healthy? From any browser: …/api/shopworks-odbc/health (+ /po-health, /contacts-health, /designs-health, /reps-health) → returns ok, minutes since last success, and the last summary.
  • Run one now: right-click → Run, or schtasks /Run /TN "\NWCA\ShopWorks ODBC Order Sync". Then read its .log.
  • See what happened: the Last Run Result column, or the task's History tab, or the tail of its .log.
  • Change a script's behavior: edit the repo master, copy to the matching C:\NWCA\… path — no task change needed.
  • Keep thumbnails flowing: keep exporting "Thumbnails Report" from ShopWorks into \\BANDIT\SWF (= C:\SWF); the metadata sync reads the newest one each run.
  • Handy script flags (run manually): -DryRun (validate, 0 writes), -SeedSnapshot (baseline after a bulk import), -CsvOut <path> (emit a CSV for a $0 bulk import).

Never-break rules for this folder

  • Only touch tasks inside \NWCA. Leave the OneDrive / Adobe / Edge / Norton / Windows tasks alone — they're app/OS-managed, get auto-recreated, and deleting them breaks sync/updates/antivirus.
  • New sync? Create it inside \NWCA, run as SYSTEM, script in C:\NWCA\… with its master in the repo bandit-agent/ folder.
  • Never run a large metadata + image batch at the same time (per-second Caspio limit). Bulk loads go through a CSV import, not the API.

How each sync works

  1. Read — the agent opens the FileMaker ODBC DSN (SWODBC, login extro/extro, read-only) and pulls a bounded query (orders/PO = rows changed since last run; contacts/designs/reps = full scan). Thumbnail images are read from the file share instead.
  2. Join what isn't on the main table — the Order Sync runs a second bounded query for the ship method, which ShopWorks keeps on the address row (Addr.ShipMethod), not on Orders. It fetches the method for just the orders in the delta (chunked id_Order IN (…)) and attaches it as an extra ShipMethod column on ORDER_ODBC — which is what lets the AE data-quality radar flag "a ship method was chosen but no ship-to address was entered".
  3. Clean — the agent applies the same normalization the old OneDrive export used (rep names, address/city/state, tiers, flags).
  4. POST — cleaned rows go in chunks to the proxy endpoint above (header x-crm-api-secret). The proxy is the only thing that writes Caspio, upserting by key (ID_Order / ID_PO / ID_Contact / ID_Customer / ID_Design / ID_Serial). It also uploads images to Box.
  5. Heartbeat — every run stamps Sync_Heartbeats so the watchdog can tell "alive but quiet" from "dead".

Each agent supports handy flags: -DryRun (validate + 0 writes), -SeedSnapshot (adopt current state as the delta baseline), -CsvOut <path> (emit a table-shaped CSV for a bulk Caspio import — the import quota, not the API budget).

Big backfills = CSV import, not the API

A Caspio CSV import runs on the Data import/export runs quota (~90/1000), not the metered Integrations API calls (500K/mo). Every agent's -CsvOut mode emits a CSV so a 25k-row backfill costs $0 in API. That's how the full thumbnail backfill was done.

Thumbnails & Box

  • Two parts. Metadata rows (design name, part #, dims) come from Erik's Thumbnails Report*.xlsx export dropped into \\BANDIT\SWF. Images come from the FileMaker container store shared as \\NCA-FS01\Thumbnails (read-only — never write it).
  • All images live on Box (folder ShopWorks Thumbnails Archive, id 400901982283). As of 2026-07-18: 26,896 thumbnails serve from Box, 0 from Caspio. The upload happens on the proxy via uploadFileToBox() using Box creds in the Heroku env — there is no Box client or token on bandit.
  • Transparent to the website. Every consumer resolves the URL as ExternalKey ? /api/files/{key} : FileUrl, so a row that points at Box just serves from Box — no app change needed.

Connecting & troubleshooting

  • Reach bandit from Erik's laptop over PowerShell Remoting: $cred = Import-Clixml "$env:USERPROFILE\bandit-cred.xml"; Invoke-Command -ComputerName bandit -Credential $cred { … } (the credential is DPAPI-encrypted, laptop-bound — re-create it after any domain password change). Or plain RDP.
  • Is a sync healthy? GET /api/shopworks-odbc/health (and /po-health, /contacts-health, /designs-health, /reps-health) return ok + minutes since last success.
  • Logs live next to each script: C:\NWCA\odbc-sync\*.log, C:\NWCA\thumb-image-sync\*.log, C:\NWCA\thumb-sync\*.log. Config for all: C:\NWCA\odbc-sync\config.json (ProxyBase, CrmApiSecret, Dsn).
  • Task inventory: use schtasks /query — the Get-ScheduledTask cmdlet returns empty over PS-remoting on this box.

Vendor rules — OnSite is LIVE production

  • Never blanket-query the ODBC DSN — always WHERE + FETCH FIRST n; unbounded scans can crash OnSite for everyone. No writes ever (order creation stays on the ManageOrders push API).
  • Never poll tightly — minutes apart, not seconds. And never two write-heavy Caspio syncs at once (see the rule at the top).
