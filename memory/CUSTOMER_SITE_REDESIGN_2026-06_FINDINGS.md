## Audit: catalog-nav

**Summary:** Product discovery on the current site is a single-page homepage app (index.html + 5 root-level JS files) where search, category browsing, and results all render in-page against the caspio-pricing-proxy /api/products/search endpoint. The machinery is functional on desktop but has one show-stopper (the entire top nav INCLUDING the search box is display:none on mobile <=768px), a hardcoded customer-facing price formula that violates the pricing-from-API rule, a nav menu where 3 of 7 items are dead ends (Marketing 404s, Sale and Resources are 'coming soon' placeholders), zero URL/deep-link state for category or search results, and a 16-category taxonomy hardcoded in JS. Backend support is substantially better than the frontend exposes: /api/products/search already supports brand/category/subcategory/color/size/price filters, sort, pagination, isTopSeller, and facets, but the UI only surfaces a Brand filter. The redesign should replace the in-page SPA with a real /catalog page with URL-driven state, API-driven categories and prices, and a shared header with drawer nav per the new nwca-2026.css system.

## Audit: tees-blueprint

**Summary:** The Custom T-Shirts storefront (LIVE v2026.06.11.1) is a 5-module frontend (pages/custom-tees.html + pages/js/custom-tees-{app,pricing,designer,calibration,shipdate}.js) backed entirely by THIS repo's server.js (not proxy routes): a channel-aware checkout route (orderSettings.channel='custom-tees' selects rebuildCtsQuote), a signature-verified Stripe webhook that sends confirmation emails, stamps idempotency/status-token into OrderSettingsJSON, and self-POSTs /api/submit-3day-order to push to ShopWorks via the proxy's /api/manageorders/orders/create. All money comes from Caspio at runtime (per-style /api/pricing-bundle?method=DTG + Service_Codes 3DT-RUSH/CTS-LTM/CTS-SHIP-FLAT/CTS-SHIP-FREE-OVER + /api/tax-rates/lookup), fail-closed, with the server authoritatively repricing and rejecting >1-cent client mismatches. Orders live in Caspio quote_sessions with QuoteID DTG{MMDD}-{rand4} and four JSON blobs (CustomerData/ColorConfigs/OrderTotals/OrderSettings) that every downstream consumer (webhook, push, emails, /order-status, admin artwork views) reads. A Hats clone is mostly a parameterization exercise â€” new channel string, new pure pricing module (CAP embroidery: /api/pricing-bundle?method=CAP + CAP-AL, 1-7/8-23/24-47/48-71/72+ tiers, LTM qty<=7, CeilDollar rounding), a caps catalog table, one-size qty UI instead of the size grid, fixed cap logo locations (front + optional back via CAP-AL) instead of free-placement platen tiers, and cap-aware designer geometry â€” but the channel string is hardcoded in ~6 scattered places that must each be extended, and several tee-specific constants (designTypeId 45, 'LTM-75' part, 16x20 envelope, PC54 Milton rush feed, 'Port & Company Core Cotton Tee' fallbacks) are buried inline.

### Key facts
- refresh=true on every quote_sessions lookup is LOAD-BEARING: the proxy caches lookups 5 min, and the checkout's pre-create uniqueness check would otherwise poison the cache with [] under the new QuoteID â€” the webhook then reads the poisoned [] and declares 'payment without record' (found live 2026-06-09). server.js fetchQuoteSessionRow L555 + checkout L2631.
- Webhook idempotency is the Caspio Status ladder, not a key: 'Processed'/'ShopWorks Failed' = terminal skip; 'Payment Confirmed' = intermediate â€” a redelivery finding it raises a Slack alert (duplicate-push risk) instead of silently skipping; a FAILED row lookup returns 5xx so Stripe retries (transient Caspio outage must self-heal). server.js L815-852.
- Confirmation emails are sent AFTER the idempotency check but BEFORE the 'Payment Confirmed' PUT, so the success-page poller always sees emailsSentAt together with a confirmed status â€” this ordering kills the browser double-send race. Browser send (custom-tees-success.js sendEmailsOnce) is FALLBACK only, gated on the stamp being absent.
- EmailJS service id is HARDCODED 'service_1c4k67j' in server.js (ORDER_EMAILJS_SERVICE) and custom-tees-success.js â€” the EMAILJS_SERVICE_ID env var points at a DIFFERENT service in the same account; do not use it. Keys: EMAILJS_PUBLIC_KEY ('4qSbDO-SQs19TbP80', also hardcoded client-side) + EMAILJS_PRIVATE_KEY (server only).
- Shipped-email dedup stamp shipEmailSentAt is merged with Object.assign over the parsed OrderSettingsJSON so it never clobbers emailsSentAt/statusToken (E2E-proven idempotent 2026-06-11). The trigger is the single tracking chokepoint POST /api/quote-sessions/:quoteId/shipstation-tracking (server.js L7457) â€” both the ShipStation webhook and the hourly backfill cron write through it. That endpoint currently has NO auth.
- Stock gate is rush-aware and fail-OPEN by design: PC54+rush gates on the Milton feed (/api/manageorders/pc54-inventory), everything else on SanMar (/api/sanmar/inventory/:style, color IS CATALOG_COLOR, totalQty pre-summed). A would-be shortage is re-confirmed on ONE fresh re-fetch before 409ing STOCK_CONFLICT; feed errors stamp stockChecked:false which becomes a '*** STOCK NOT VERIFIED ***' banner in the ShopWorks push note. BUT the SanMar proxy route caches ~5 min with no refresh param, so the 'fresh' re-check can still be 5-min stale. Client-side: Milton rush recheck is fail-CLOSED, SanMar standard recheck fail-open.
- A color the stock feed doesn't name at all is SKIPPED by ctsStockConflicts (fail-open â€” naming mismatch must never read as sold-out); means any CATALOG_COLOR drift between catalog table and feed silently disables stock tracking for that color.
- Server reprice re-derives the price tier from sanitized placement dimensions (clamped to 16x20) via the same pure locationForArtSize() rule the browser uses â€” client location codes are advisory; a doctored payload cannot buy jumbo art at left-chest rates. Size whitelist from the SERVER-fetched bundle is equally load-bearing: an unknown client size key would inflate the tier while pricing $0.
- Baked LTM (UberPrints model): config.bakeLtm folds floor((CTS-LTM/qty)*100)/100 per piece into every unit price â€” exact to the penny because the order LTM line was already perPiece*qty. ltmFee=0 everywhere by design; audit trail in orderSettings.ltmBaked{perPiece,total}. The tier row's LTM_Fee>0 gates WHEN; the Caspio CTS-LTM code ($25) gates HOW MUCH (overrides the builder's $50). Sub-24 storefront totals intentionally diverge from the internal builder.
- CTS-SHIP-FREE-OVER's SellPrice field holds the THRESHOLD dollars ($100), not a price â€” a semantic trap when creating the hats equivalents. Both threshold codes fail checkout CLOSED if missing/inactive in Caspio (code() throws).
- QuoteID = DTG{MMDD}-{rand4} (DTG prefix deliberately chosen so storefront orders land in Quote Mgmt next to internal DTG builder quotes). Uniqueness = up-to-4 random retries against a live lookup; the check is fail-OPEN (lookup error accepts the candidate).
