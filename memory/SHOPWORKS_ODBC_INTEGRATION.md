## Driver / DSN tech facts (Claris)

- xDBC port **2399, fixed**; FMS-side needs ODBC/JDBC sharing on + account with `fmxdbc` extended privilege; listener = separate `fmxdbc_listener` process (known to wedge/die — treat "connection suddenly dead" as expected; restart via FMS Admin Console = disconnects ALL ODBC clients incl. shipping).
- **Driver bitness must match the app, not the OS.** 32-bit dll → `SysWOW64`, 64-bit → `System32`. Both ODBC admins are literally `odbcad32.exe` — folder decides. Wrong-bitness DSN = `IM002`/`IM014`. Check what the ShopWorks installer laid down BEFORE picking runtime; if 32-bit-only, use 32-bit Python/Node.
- Claris version-pairs client driver ↔ server ("driver for the current version is not compatible with earlier versions") — that's why the OLD vendor installer is required; do NOT substitute a newer Claris driver.
- DSN options that matter: ✅ "Describe text fields as long varchar" (else >255-char text comes back EMPTY), ✅ UTF-8, ✅ "Save long-running queries to a log file". Use a **System DSN** so Task Scheduler jobs can see it.

## FileMaker SQL dialect gotchas (query style guide)

- Row limit: `OFFSET n ROWS FETCH FIRST n ROWS ONLY` (no LIMIT/TOP). Delta-sync on an indexed modification-timestamp beats OFFSET paging.
- Dates via ODBC: `{d '2026-01-01'}` / `{ts '2026-01-01 14:35:10'}` (SQL-92 `DATE '...'` also OK). **`BETWEEN` on dates is pathologically slow — use `>= AND <=`.**
- Empty string = NULL in FileMaker (`col = ''` matches nothing → use `IS NULL`). String compare is CASE-SENSITIVE; `LOWER()` fix kills index use. No TRUE/FALSE — use 1/0. No RIGHT/FULL OUTER JOIN (rewrite as LEFT).
- **Unstored calculation/summary fields are per-row-evaluated, non-indexable, and ShopWorks schemas are full of them** — one in a WHERE (or even SELECT list) turns sub-second into minutes and spikes FMS CPU for OnSite users. Explicit column lists only, never `SELECT *`.
- Assume queries are SERIALIZED through the listener (fixed ~FM19; OnSite embeds older FMS) → ONE connection, sequential queries, connect/login timeouts set, retry-once-on-dead-connection.

## ✅ Ship method (Addr.ShipMethod) joined into ORDER_ODBC 2026-07-20 — powers the AE data-quality "no ship-to" check

- **WHY**: the AE Mission Control data-quality radar wanted to flag "a ship METHOD was chosen (e.g. UPS Ground) but the ship-to address block is blank" literally, instead of the `cur_Shipping>0` dollar approximation. **KEY SCHEMA FACT (verified against `shopworks-odbc-schema-catalog.txt`)**: the ShopWorks **`Orders` table has NO ship-method column** — `ShipMethod` (varchar) lives on the **`Addr`** (per-order address) table, linked by `Addr.id_Order`; also on `PO`; `OrdTyp.pref_ShippingMethod` is only an order-type default. So there was nothing to add to the `FROM Orders` SELECT — the method has to be JOINED in from Addr.
- **34th ORDER_ODBC column added**: `ShipMethod` (Text/STRING, nullable) via idempotent `caspio-pricing-proxy/scripts/add-order-odbc-shipmethod.js --apply` (ran 2026-07-20 → table went 46→47 fields). `STRING`=Text255; short labels fit.
- **Agent** `bandit-agent/sync-orders.ps1`: after reading the changed Orders, runs a **second bounded query on the SAME connection** — `SELECT id_Order, ShipMethod FROM Addr WHERE id_Order IN (<delta ids, chunked 200>) AND ShipMethod IS NOT NULL` — builds an `id_Order→ShipMethod` map (first non-blank wins if multiple ship-to rows) and attaches `ShipMethod` to each order row before POSTing. `Addr.id_Order` is a plain stored/indexed field → cheap IN filter; delta keeps the id list short. Sequential (never a JOIN in the delta query) per the vendor "one connection" rule.
- **Route** `shopworks-odbc-sync.js`: `ShipMethod` added to `ODBC_FIELDS` (whitelist) → written on each order upsert like any other overwrite field.
- **Consumer** `ae-dashboard.js` `dqOrderIssues`: `ShipMethod` added to the DQ `q.select`; blank ship-to + a **real CARRIER** `ShipMethod` ⇒ **err** `ship method "<x>" chosen but NO ship-to address`. **`cur_Shipping>0` DEMOTED** to a secondary fallback (covers rows synced before ShipMethod backfilled, and shipping charged with the method left blank); truly-blank method+address+charge = pickup **warn** (unchanged). Frontend `dqChips` already `esc()`-escapes the text (no XSS from the method value).
- **🔴 CRITICAL: pickup exclusion (`isNonShippingMethod`) is MANDATORY** — live probe (2026-07-20) showed `Addr.ShipMethod` is populated on ~almost every order and **"Customer Pickup" is the single MOST COMMON value** (39/55 recent; also "Will Call"-style + placeholders "Need Ship Method"/"ask when done"). Pickups routinely have a blank ship-to → without the guard, EVERY pickup would false-err. `isNonShippingMethod()` excludes the pickup family (substring `pickup`/`pick up`/`will call`) + a placeholder set; only real carriers (UPS Ground, UPS, **Ground Shipping**, FedEx…) trigger the err. **Simulated against live data**: of 122 recent orders, 16 pickup+blank-ship correctly ignored, **5 real carrier+blank-ship flagged** (ords 142385/142392/142397/142401/142402 = genuine "UPS/Ground picked, no address" gaps). Erik-tunable set as new placeholder values appear.
